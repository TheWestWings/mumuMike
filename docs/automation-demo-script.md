## 林雅琼台词：组长 / CI/CD 负责人

大家好，我是本组组长兼 CI/CD 负责人，负责本次自动化测试体系设计、GitHub Actions 流水线配置和最终演示整合。

我们这次的目标，不是简单补几个测试用例，而是把前两次评审形成的质量问题接入流水线。也就是说，代码评审发现的问题，要通过自动化测试持续验证；测试评审设计的主流程，要通过 API 回归反复执行。

首先看整体设计。我们的系统是 Spring Boot 后端、Vue 后台管理端和微信小程序。根据任务要求，我把自动化分成两层：第一层是研发侧代码级测试，主要用 JUnit 验证核心业务逻辑和权限边界；第二层是客户侧 API 回归测试，用 pytest 和 requests 模拟真实用户流程。GitHub Actions 里先执行代码级测试，只有第一层通过，才会启动 API 回归测试。

这里我打开 `.github/workflows/automated-tests.yml`。可以看到流水线有两个 job：第一个是 `backend-code-test`，运行 `mvn test` 并上传 JUnit 报告；第二个是 `api-regression`，它依赖第一个 job 成功后才执行。第二个 job 会启动 MySQL 服务，导入 `test-data/schema.sql` 和 `test-data/seed.sql`，再启动后端服务并执行 API 回归测试。

这部分设计对应任务里的质量门禁要求：如果代码级测试失败，API 回归不会继续跑；如果 API 回归失败，PR 检查也不会通过。后面三位同学会分别展示每一层测试的具体内容。

## 杨仕强台词：模块 A 订单交易与支付闭环代码级测试

大家好，我负责模块 A，也就是订单交易与支付闭环模块的代码级自动化测试。这个模块在代码评审中暴露过三个风险：订单归属依赖前端传入的 `userId`，订单商品状态接口权限不足，微信支付接口仍处于开发模式。

我先展示订单相关的修复点。现在创建订单时，后端会从 `Authorization` token 中解析当前登录用户，并把订单归属绑定到这个用户，而不是信任前端传入的 `userId`。这样可以避免顾客通过篡改请求体，把订单挂到其他用户名下。

接着看 JUnit 测试。`OrderServiceImplTest` 验证创建订单时，订单明细会绑定数据库生成的订单 id；`OrderControllerSecurityTest` 验证普通用户不能调用订单商品状态修改接口，同时验证创建订单时以后端登录用户为准。

我这里运行或展示 `backend-code-test` 报告，可以看到订单相关测试全部通过。这个测试点对应测试计划中的小程序下单、订单详情核对和非本人订单边界，也对应代码评审中“订单归属错误”和“越权修改订单状态”的整改。

支付部分我们这次不接真实微信商户号，而是做开发模式接口校验。`WxPayControllerTest` 验证 `/wx/pay/create` 能返回稳定的模拟支付参数，`/wx/pay/query/{orderId}` 能返回固定结构。这样既覆盖支付接口契约，又避免现场演示依赖外部支付环境。

## 郑子龙台词：模块 B 菜品与菜单管理代码级测试

大家好，我负责模块 B，也就是菜品与菜单管理模块的代码级自动化测试。这个模块的核心目标是保证后台商品管理安全，同时保证小程序菜单只展示有效数据。

代码评审中，这个模块有三个风险：商品管理接口没有角色限制，修改商品时图片参数是必填，公开商品列表可能返回下架商品。针对这些问题，我们做了三类修复：第一，商品新增、修改、删除、上下架接口都补充 `@Role(role = {0, 1})`；第二，商品编辑时图片改为可选，不上传新图片也能只修改文字信息；第三，公开给小程序的菜单接口只保留 `getProductVOList`，并且它只查询启用分类和上架商品。

我这里展示 `ProductControllerSecurityTest` 和 `ProductTypeControllerSecurityTest`。这两个测试用反射检查管理接口是否真的添加了权限注解，避免后续有人改代码时把权限漏掉。

再看 `ProductServiceImplTest`。这个测试验证菜单 VO 查询时，分类条件必须是 `status = 1`，商品条件也必须是 `status = 1`。也就是说，下架商品和停用分类不会进入客户侧菜单。

这些自动化测试对应测试计划中的后台新增商品、商品上下架、分类停用和菜单刷新，也对应代码评审中“普通用户可能调用商品管理接口”和“公开查询可能返回下架商品”的问题。

## 祝浩然台词：模块 C 权限测试与 API 回归测试

大家好，我负责模块 C 相关权限测试和客户侧 API 回归测试。模块 C 的重点是登录、token、角色权限和用户状态。

代码评审中，这个模块有一个典型问题：部分接口直接对 `Authorization` 请求头执行 `substring(7)`，如果没有 token 或者 token 格式不对，就可能出现 500 异常。现在我们新增了统一的 `AuthUtil`，先判断请求头是否存在、是否以 `Bearer ` 开头，再校验 token 是否有效。`UserControllerAuthTest` 就是验证无 token 和错误 token 前缀时，系统返回 401，而不是服务端异常。

下面看 API 回归测试。`api-tests/test_regression.py` 用 pytest 和 requests 模拟客户侧真实流程：先验证正常用户登录成功、禁用用户登录失败；再调用公开菜单接口，断言只出现启用分类和上架商品；然后顾客提交订单，断言订单归属是当前登录用户；管理员把订单状态改为完成后，顾客消息列表里应该出现取餐提醒；最后普通用户尝试调用商品、分类和订单管理接口，预期是 401 或 403。

这部分对应测试计划里的登录注册、角色权限、菜单展示、下单、订单处理和消息提醒主流程。它和前面的 JUnit 不一样，JUnit 更偏代码级，API 回归更接近真实用户视角。

在 GitHub Actions 中，API 回归会启动临时 MySQL，并导入测试数据。这样测试结果不依赖我们本机数据库，现场和老师复查时都能复现。

## 林雅琼收尾台词

最后我来总结本次自动化测试闭环。

第一，我们完成了分层自动化：JUnit 负责研发侧代码级测试，pytest 负责客户侧 API 回归测试。第二，我们把代码评审中的问题转成了自动化断言，包括订单归属、商品管理权限、token 边界和菜单过滤。第三，我们通过 GitHub Actions 把这些测试接入了 CI 流水线，形成了提交代码后自动执行、测试失败阻止合并的质量门禁。

大家可以看这里的 PR Checks，`backend-code-test` 和 `api-regression` 都通过后，代码才能合并到主分支。如果其中任何一个失败，GitHub 会直接标红，测试报告和 app 日志也会作为 artifact 保留下来，方便定位问题。

所以这次成果不是一次性的演示脚本，而是一个可以持续运行的自动化质量保障流程。以上是我们组的展示，谢谢大家。

## 备用：个人 5 分钟压缩版

大家好，我是本组组长兼 CI/CD 负责人。我们这次自动化测试的目标，不是简单补几个测试用例，而是把前两次评审形成的质量问题真正接入流水线。

首先看整体设计。我们的系统是 Spring Boot 后端、Vue 后台管理端和微信小程序。根据任务要求，我把自动化分成两层：第一层是研发侧代码级测试，主要用 JUnit 验证核心业务逻辑和权限边界；第二层是客户侧 API 回归测试，用 pytest 和 requests 模拟真实用户流程。GitHub Actions 里先执行代码级测试，只有第一层通过，才会启动 API 回归测试。

第二部分是测试来源。我们的测试用例不是凭空写的，而是来自两份材料：一份是测试评审会里的核心业务用例，另一份是代码评审会确认的缺陷。比如商品管理接口缺少权限、订单状态接口可能被普通用户调用、Authorization 请求头直接 substring 导致异常，这些都被转成了自动化断言。

第三部分是流水线。大家可以看 GitHub Actions，这里有两个 job。`backend-code-test` 会执行 Maven 测试并上传测试报告；它成功后，`api-regression` 会启动临时 MySQL，导入测试 SQL，启动后端服务，再执行 API 回归测试。这样每次提交代码，都能自动验证核心业务是否被破坏。

第四部分是质量门禁。我们在 GitHub 的 PR 检查里要求两个 job 必须通过，才能合并到 main。也就是说，如果有人改坏了权限控制、订单归属或菜单过滤，流水线会直接失败，PR 不能合并。这就把“代码评审发现问题、测试评审设计用例、自动化持续回归”串成了一个闭环。

最后总结一下：本次自动化覆盖了登录权限、菜单展示、下单、订单处理、消息提醒和开发模式支付接口。它的价值是让测试从一次性的人工检查，变成每次提交都会自动执行的质量保障机制。以上是我的展示，谢谢大家。
